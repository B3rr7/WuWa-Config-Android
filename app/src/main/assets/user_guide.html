1. Introduction

WuWaConfig is an Android tool that reads, optimizes, and deploys configuration files for Wuthering Waves (global version, com.kurogame.wutheringwaves.global). It works with the 5 INI files the game uses:

FilePurpose
Engine.iniCVar settings (graphics, performance, shadows, anti-aliasing, etc.)
DeviceProfiles.iniDevice-specific GPU profiles and per-device CVars
GameUserSettings.iniResolution, VSync, FPS cap, scalability groups
Scalability.iniResolution quality, view distance, shadow/texture/effects quality
Hardware.iniHardware profile binding, frame pacing, Anisotropy, mip bias

What it can do:

  • Analyze your device's Client.log to extract GPU, RAM, FPS, thermal data, OOM events
  • Score your device with SmartBrain (0-100 scale) and recommend the optimal preset
  • Generate optimized INI configs from 8 presets (Potato→Cinematic) or per-device tuning
  • Deploy configs via ADB, Shizuku, Root, or SAF
  • Backup configs before applying, with per-file selection
  • Compare performance before/after with Deploy History
  • Track gacha pity, view player profile, read battle stats from Client.log

Package: com.wuwaconfig.app
Min SDK: 26 · Target SDK: 34
Language: Kotlin · UI: Jetpack Compose + Material 3

3. Access Methods — How to Connect

The app needs to read/write game config files at:

/storage/emulated/0/Android/data/com.kurogame.wutheringwaves.global/files/UE4Game/Client/Client/Saved/Config/Android/

Choose your connection method from the Home screen by tapping the method chip (cycles ADB → Shizuku → Root → SAF).

3.1 ADB (Wireless Debugging) — Recommended

Best for: Non-rooted users. The app implements the ADB wire protocol directly — no external ADB binary needed.

Setup:

Option A — In-app (no PC):

  1. Go to Developer Options on your phone
  2. Enable Wireless Debugging
  3. Tap Wireless Debugging → Pair device with pairing code (if first time)
  4. Back in the app, tap Connect — it auto-scans ports 37000–44000
  5. Accept the RSA fingerprint prompt on your phone

Option B — PC commands:

Requirements: USB Debugging ON in Developer Options, phone connected via USB.

Enable:

adb tcpip 5555
adb connect 192.168.x.x:5555

Disable:

adb disconnect 192.168.x.x:5555
adb usb

If multiple devices: adb -s 192.168.x.x:5555 usb

No PC? Reboot your phone — ADB TCP mode clears on boot. The app's Disconnect button only closes the app socket, not the phone port.

Then tap Connect in the app — it finds port 5555 automatically.

Troubleshooting:

  • On Android 16+, you may need to re-pair after reboot
  • The app auto-regenerates RSA keys if connection is rejected
  • If auto-scan fails, tap the ADB chip and enter IP:port manually

3.2 Shizuku

Best for: Non-rooted users with Shizuku installed. Faster than ADB.

Setup:

  1. Install Shizuku from the Play Store
  2. Start the Shizuku service (follow Shizuku's in-app instructions)
  3. In WuWaConfig, cycle to Shizuku mode
  4. Tap Permit → Connect

3.3 Root

Best for: Rooted devices (Magisk, KernelSU, APatch).

Setup:

  1. Cycle to Root mode
  2. Tap Test Root — grant superuser access in your root manager
  3. Tap Connect

3.4 SAF (Storage Access Framework)

Best for: Quick one-off edits. Limited — no shell access, so config generation and some features are unavailable.

Setup:

  1. Cycle to SAF mode
  2. Tap Pick Dir
  3. Navigate to the game's Config/Android folder
  4. Tap Allow

4.6 INI Editor

Edit any of the 5 monitored INI files directly on-device:

  1. Tap INI Editor button → file list of all 5 INI files
  2. Tap a file → reads current content from device into a monospace text editor
  3. Edit CVars freely — add, change, or remove any values
  4. Tap Save → pushes to device, refreshes KuroConfigMonitor hash
  5. Success popup confirms write + hash refresh

Auto-sync on open: When you enter the INI Editor, the app reads all 5 INIs, computes their MD5s, and refreshes the hash file if any mismatch is detected — prevents desync before you start editing.

4.7 Clean Config Files

Replaces the old Delete Config Files. Instead of deleting INIs entirely, strips CVar values while preserving critical structure:

  • Clean — strips all CVar values. Engine.ini keeps only [Core.System] paths. Other INIs keep section headers only. Game regenerates defaults on next boot.
  • Delete — removes selected INI files entirely via rm -f. Game creates fresh versions on next launch.

Both actions use the hash snapshot + reconcile pattern: saves hash file before modifying, compares afterward to detect concurrent game access, and always recomputes from actual files.

Benchmark Tuner

Auto-tune wizard that preserves state across app restarts:

  1. Tap Auto-Tune → deploy current preset
  2. Go Play dialog appears → play the game for ~30 seconds
  3. Tap Continue → reads logcat, parses FPS
  4. If FPS ≥ target → preset stays or bumps up. If FPS < 85% of target → preset drops down. Options adjust based on gap size.
  5. Up to 5 rounds total. State saved to benchmark_tuner_state.json — survives app close.

5.3 Generating & Deploying

  1. Configure all options
  2. Tap Generate — creates all selected INI files
  3. ReviewTune screen opens — a generated-config reviewer/editor/deploy view with a monospace text editor
  4. Tap Deploy to Device — pushes files, uses hash snapshot+reconcile to detect concurrent game access, refreshes KuroConfigMonitor hash
  5. Verify — app pulls Client.log and cross-references deployed CVars

Hash integrity: Before every deploy, edit, or clean operation, the app snapshots the current hash file. After the operation, it compares the snapshot against the live file. If the game modified the hash during the operation (concurrent access), the app logs a warning and reconciles. If unchanged, it safely updates. All paths verify actual INI MD5s match the stored hashes.

12. Deploy History

Access from Home screen → Deploy History card or from History screen.

Tracks each config deployment and lets you compare outcomes after gameplay.

12.1 How It Works

  1. Baseline — when you deploy a config, the app saves a DeployRecord
  2. Compare — tap Compare Now on a record
  3. App reads fresh Client.log and computes Δ vs baseline:
    • FPS change
    • Thermal events change
    • OOM count change
    • Frame drop change
  4. Results color-coded: green = improvement, red = regression
  5. If record was Advanced per-device tuned, a Retune & Deploy button appears:
    • Stable (±5 FPS, 0 OOM, ≤2 thermal, ≤5 drops) → no change
    • OOM increased → force potato-level
    • Degraded → screen×0.75, shadow-2, detail-1, SSR-1
    • Improved → screen×1.15, shadow+1, detail+1, SSR+1
    • Generates new config, deploys, saves new record — repeat until stable

12.2 Settings

Toggle in Settings → Generation & History → Deploy History. Default: ON.

12.3 Managing Records

  • Delete single record — tap the delete icon (trash) on any record card
  • Clear all — tap Clear All button at top of list, confirm in dialog
  • Automatic cap: maximum 20 records kept (oldest evicted first)

  • Device evaluation tier, texture LOD bias, draw distances, imposter scale, ISM distances, foliage cull, FPS unlock

  • [ScalabilityGroups] — sg.* values
  • [/Script/Engine.GameUserSettings] — VSync, resolution, FPS cap, HDR

Optional files (OFF by default): resolution quality, quality levels, hardware profile, frame pacing.

13.5 Core.System Paths

Before generation, the app reads existing Engine.ini from device and extracts [Core.System] paths (~40 content paths). If no existing config, built-in defaults are used.

13.6 Post-Processing

  1. Import from log — CVars from Client.log added
  2. CVar overrides — user's key=value pairs
  3. CVar optimization — redundant/unknown CVars commented out
  4. Deduplication — duplicate keys removed (last wins)
  5. Forbidden CVar stripping — 31 Kuro-restricted CVars removed

13.7 Advanced Per-Device Tuning

When ON, the fixed preset lookup is replaced by CvarOptimizer:

  • GPU tier detection (flagship/high/mid_high/mid/mid_low/low/unknown)
  • Per-field rules: screen%, shadow, SSR, mipbias, streaming, view distance, foliage LOD, detail, LOD bias, grass cull
  • Hard-limited: OOM ≥ 1 OR (thermal ≥ 3 AND low-tier GPU) → minimum settings
  • Constrained: thermal ≥ 3, texture errors ≥ 5, or RAM < 6GB → 1-2 tiers lower

Self-tuning feedback loop:

  1. Deploy with Advanced Gen ON → saves OptimizedProfile in history
  2. Play game → new Client.log created
  3. History → Compare Now → pulls log, computes Δs
  4. Retune & Deploy — adjustProfile() adjusts based on Δs, generates new config, deploys
  5. Repeat until stable

13.8 File Push Mechanism

  1. Content base64-encoded
  2. Split into printf '%s' chunks (~4000 bytes)
  3. Written to /data/local/tmp/wuwaconfig_{ts}_{rand}.b64
  4. Decoded via base64 -d
  5. Temp file deleted (rm -f /data/local/tmp/wuwaconfig_*.b64 before every push)
  6. MD5 verified
  7. Retried once on failure

Hash file safety (7 mechanisms):

  1. Adaptive hash patching — preserves ModifyCount, unknown fields (Signature=, Version=), formatting, and non-monitored sections
  2. Atomic write — pushes to .new temp file, then mv renames — crash mid-write never corrupts live file
  3. Zero stale temp files — cleans up base64 artifacts before every push
  4. ModifyCount never incremented — reads existing count, preserves it as-is
  5. Snapshot + Reconcile — before any config op, snapshots hash file. After op, compares vs current to detect concurrent game access. Recomputes actual INI MD5s regardless.
  6. Auto-sync on Connect — runs syncConfigHashes() every time you connect, auto-fixes desync
  7. Auto-sync on Editor open — reads all 5 INIs, compares MD5s against hash file, refreshes if mismatched

15. CVar System

15.1 What Are CVars?

Console Variables are Unreal Engine 4's runtime configuration system. Each CVar is a key=value pair controlling a specific engine behavior.

15.2 CVar Database

FileEntriesDescription
libUE4_cvars.txt5237Every registered CVar in Kuro's UE4.26 fork
config_monitor_cvars.txt693CVars tracked by KuroConfigMonitor hash
config_monitor_values.txt693Game's default values for monitored CVars

Used for: isKnown(), isMonitored(), gameDefault(), categorize() (18 categories), optimizeIniText().

15.3 Forbidden CVars

31 CVars the game actively strips/ignores: r.Streaming.Boost, r.ScreenPercentage, r.AFME.Enable, r.Streamline.DLSSG.RetainResourcesWhenOff, r.MFRC.Enable, and more.

15.4 CVar Priority Chain

SetBySystemSettingsIni > SetByDeviceProfile > SetByProjectSetting > SetByScalability

Engine.ini [SystemSettings] always wins.

16. FAQ & Troubleshooting

16.1 Common Issues

"Cannot connect via ADB"

  • Ensure Wireless Debugging is enabled in Developer Options
  • Check your phone and PC are on the same network
  • Restart Wireless Debugging (toggle off/on)
  • App auto-scans ports 37000-44000 — enter manually if needed

"Connection failed: process hasn't exited" (Shizuku)

  • Known Shizuku race condition. App retries after 5s. Restart Shizuku if it persists.

"Client.log too large"

  • App handles 100MB+ files with adaptive chunking (up to 30×1MB chunks, gzip). Ensure backend supports dd (SAF does not).

"No Convene URL found"

  • Open Convene History in-game first, wait, try again
  • Use Poll in Background

"Config deploy failed"

  • Check game directory exists, ensure write access, try different method

"Video background opacity doesn't work"

  • Known ExoPlayer/SurfaceView limitation. App overlays a black box with inverse opacity.

16.3 Client.log Decryption

The game's Client.log is encrypted and decrypted automatically by the app.

16.4 Log Reading for Large Files

For files >1MB, adaptive chunking: 1 chunk per 5MB, min 5, max 30, each 1MB via dd, gzip-compressed. ~20% coverage of a 100MB file, ~6MB wire transfer.

16.5 Data Safety

The app never sends data off-device. All operations are local. No telemetry, no analytics, no internet except gacha API calls.
